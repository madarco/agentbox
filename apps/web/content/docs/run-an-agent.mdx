title: Run an agent
description: Launch Claude Code, Codex, or OpenCode in a box and attach, detach, or fork sessions

AgentBox runs **Claude Code**, **OpenAI Codex**, and **OpenCode** inside a box, in a detachable tmux session — so you can leave the agent working, detach, and reattach later.

The three agent verbs (`claude`, `codex`, `opencode`) each create the box if it doesn't already exist, start the agent, and attach you. They share all `agentbox create` flags — see [CLI commands](/docs/cli) — plus session-resume and attach controls. This page assumes you've read [core concepts](/docs/core-concepts); the project-teleport mechanics live in [teleport a project](/docs/teleport-a-project).

<Callout title="NOTE">
`<box>` is optional almost everywhere — it defaults to the single box for the current project. Otherwise pass its short index (`1`, `2`, …), name, or id prefix.
</Callout>

## Launch an agent

`agentbox claude` is the one-liner: it creates a box (snapshotting the current directory), starts Claude Code in a detached tmux session, and attaches you. `agentbox codex` and `agentbox opencode` have the identical shape.

```bash
agentbox claude                       # create a box + start Claude Code, then attach
agentbox claude -n alpha              # name the box
agentbox claude -- --model sonnet     # pass extra args through to claude
```

Name a box with `-n`, target another directory with `-w`, and put pass-through args after `--`. The first run pulls or builds the `agentbox/box:dev` image (~1 GB, one-time) and offers a setup wizard that installs project deps and your dev server. See [local Docker](/docs/local-docker) for the image and [services and tasks](/docs/services-and-tasks) for the wizard.

```console
$ agentbox claude
log: ~/.agentbox/logs/claude.log
Starting Claude in a box...
> Run setup wizard? › Yes
  ...box ready...
attaching — Control+a d to detach, leaves claude running
```

## Detach and reattach

The agent runs in a detachable tmux session whose prefix is **`Control+a`** (not the usual `Ctrl-b`). Press `Control+a` then `d` to detach — the agent keeps running in the box.

`Control+a` also opens an actions menu: `c` (VS Code / Cursor), `s` (noVNC screen), `u` (box URL), **`t` (a fresh shell in the *same* box, opened in a new tab)**, and **`k` (destroy the box, after a confirmation)**. The shell tab opens as a cmux surface in the current workspace, a new tmux window, or a new iTerm2 tab depending on your terminal.

Reattach with the `attach` subcommand, or resync your host config and reattach with `start`.

```bash
# detach from inside the session: press Control+a then d
agentbox claude attach 1              # reattach to box 1's Claude session
agentbox claude start mybox           # start a session (resyncs host ~/.claude)
```

`attach` starts a session if none is running and auto-unpauses a stopped box, but **never resyncs your host `~/.claude`**. `start` does the resync — use it when new skills, plugins, or a fresh login should flow into the box. For `--attach-in`, `--no-attach`, and the other attach controls, see [CLI commands](/docs/cli).

<Callout type="warn" title="HEADS UP">
`attach` never resyncs your host `~/.claude` (skills, plugins, login). After installing a new skill or signing in on the host, use `claude start` (or a fresh `agentbox claude`) so the box picks it up.
</Callout>

## Switch agents

Codex and OpenCode follow the identical flow — `agentbox codex` / `agentbox opencode`, each with `attach`, `start`, and `login` subcommands.

```bash
agentbox codex                        # OpenAI Codex in a box
agentbox opencode                     # OpenCode in a box
agentbox dashboard                    # switch between running boxes/agents
```

Auto-approve is on by default because the box is isolated from your host; pass `--no-dangerously-skip-permissions` to keep the agent's own approval prompts. Each agent shares an identity volume across boxes by default — opt out per box with `--isolate-claude-config` (and the codex/opencode equivalents). See [configuration](/docs/configuration) and [CLI commands](/docs/cli).

You can run different agents in different boxes for the same project and switch between them with `agentbox dashboard`. Claude and Codex resume sessions; OpenCode does not yet (resume starts a fresh box).

<Callout title="WHY">
Auto-approve is safe to default on because the box can't touch your machine — it's isolated from the host. The agent works without stopping to ask, and you keep your laptop clean.
</Callout>

## Fork a session

The fastest way to fork is from **inside** the agent: run the **`/agentbox [provider]`** skill from Claude Code, Codex, or OpenCode. It teleports your *current* session — the full conversation, **and** the active plan if you're in plan mode — into a fresh box and resumes it there, so you can branch off an isolated copy without losing context or re-explaining yourself. The provider argument is optional (`/agentbox`, `/agentbox hetzner`, `/agentbox vercel`, …) and defaults to local Docker.

Under the hood the skill calls `agentbox fork`, which you can also run directly on the host. It opens the box in a new terminal tab under iTerm, tmux, or cmux; otherwise it runs in the background. The skill, and the other AgentBox-provided skills, are covered in [AgentBox Skills](/docs/skills).

```bash
agentbox fork                         # fork the newest host Claude session into a new box
agentbox fork --agent codex           # fork a Codex session instead
agentbox fork --provider hetzner      # fork into a cloud box
agentbox fork --name review           # name the box
```

`--agent` is `claude` (default), `codex`, or `opencode` (OpenCode starts a fresh box since resume isn't supported). `--provider` selects `docker` (default), `daytona`, `hetzner`, or `vercel`. `--plan` carries an active Claude Code plan so the box opens in plan mode, ready to resume. For `--session`, `--carry`, and the other flags, see [CLI commands](/docs/cli).

<Callout type="warn" title="HEADS UP">
`agentbox fork` runs on the host only — it teleports a host session into a fresh box, so it refuses when `AGENTBOX_RELAY_URL` is set (inside a box). Box→box fork isn't supported yet.
</Callout>

`-c/--continue` and `--resume <id>` also exist directly on `agentbox claude` / `agentbox codex` to teleport the host session in at create time — see [teleport a project](/docs/teleport-a-project).

## First-run auth

Credentials are handled on the host and seeded into every box (macOS Keychain doesn't transfer into containers). On the first `agentbox claude` with no credentials, AgentBox offers an interactive sign-in that's saved and reused by every future box. Skip it with `-y` (or in CI) and claude will prompt you to `/login` inside the box.

```bash
agentbox claude login                 # sign in once, reused by every box
agentbox codex login                  # device-auth flow for Codex
agentbox opencode login -- --provider anthropic
```

```console
$ agentbox claude
? Sign in with your Claude subscription? (saved and reused by every box) › Yes
✔ Signed in with your Claude subscription — saved for future boxes.
```

Setting `ANTHROPIC_API_KEY` / `CLAUDE_CODE_OAUTH_TOKEN` (Claude) or `OPENAI_API_KEY` (Codex) on the host forwards it into the box and skips the sign-in offer. Host `~/.claude` stays the source of truth — every `create` / `claude` syncs it in, so skills, plugins, MCP, and login flow into the next box. Cloud providers capture the login to `~/.agentbox` and push it in on create — see [hetzner](/docs/hetzner), [daytona](/docs/daytona), and [vercel](/docs/vercel).

<Callout title="TIP">
Sign in once on the host. Install a skill or sign in there, and the next box picks it up automatically. See [configuration](/docs/configuration) and [sync and git](/docs/sync-and-git).
</Callout>
