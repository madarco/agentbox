`agentbox create` teleports your project â€” git history and all â€” onto its own branch inside an isolated box, while your host checkout stays exactly as it was. The box gets a full, writable copy of the repo; your machine keeps the credentials.

See [core concepts](/docs/core-concepts) for the box / worktree / branch model. If you want to provision a box _and_ launch an agent in one step, that's [run an agent](/docs/run-an-agent) â€” `agentbox claude`, `codex`, and `opencode` do everything `create` does first.

## Create a box

From your project directory, `agentbox create` provisions one isolated box â€” a Docker container locally, or a remote sandbox with `--provider`. It detects git repos (the root plus each first-level subdirectory, for monorepos), picks a fresh per-box branch, captures your uncommitted work, starts the box, and seeds `/workspace`.

```bash
agentbox create
```

The default box name is `<workspace-basename>-<id>`. Override it with `-n, --name`:

```bash
agentbox create -n my-feature
```

To provision in the cloud, pass `--provider` and follow the setup on the provider page ([hetzner](/docs/hetzner), [daytona](/docs/daytona), [vercel](/docs/vercel); local is [docker](/docs/local-docker)):

```bash
agentbox create --provider hetzner
```

Later you refer to a box by index (`1`, `2`), name, or id prefix. See [CLI commands](/docs/cli) for all flags.

<Callout title="TIP">`agentbox create` only provisions the box. To create a box **and** launch an agent in one step, use `agentbox claude`, `agentbox codex`, or `agentbox opencode` â€” see [run an agent](/docs/run-an-agent).</Callout>

## How your files get in

On local Docker, the box's `/workspace` is a **git worktree** on its own branch (`agentbox/<box-name>`), backed by your host's `.git/`, so commits made inside the box land on your host immediately. No working copy is written to the host â€” `/workspace` and all build artifacts (`node_modules`, `.next`, `target`, `.venv`) live only in the container, survive pause / stop / start, and are wiped on `destroy`. For monorepos, each detected repo gets its own branch.

Cloud providers can't bind-mount your `.git`. Instead the workspace is seeded from a host `git bundle` (plus your uncommitted work â€” see below), and push / pull flow through the host relay.

See [checkpoints and pausing](/docs/checkpoints-and-pausing) for warm starts, and [sync and git](/docs/sync-and-git) / [access your box](/docs/access-your-box) for getting work back out.

<Callout type="warn" title="HEADS UP">The box's `/workspace` is **not** a folder on your machine â€” it lives inside the container. Use `agentbox open` or `agentbox download` to bring files back to the host. See [access your box](/docs/access-your-box) and [sync and git](/docs/sync-and-git).</Callout>

## Carry-over

Your uncommitted work comes along automatically â€” both tracked-but-uncommitted changes and untracked files. The agent starts from exactly the state you'd see in `git status`, on a fresh branch, fully isolated from your host checkout.

One boundary: **gitignored files are not carried.** That's what `--with-env` and the `carry:` block are for (next section).

<Callout title="WHY">Carry-over is non-destructive. `git stash create` never alters your host working tree or stash list â€” your machine looks exactly the same after `create` as before.</Callout>

## Environment files

Because gitignored files are excluded from carry-over, secrets and local env files don't reach the box by default. That's deliberate.

`--with-env` copies host env/config files (`.env*`, `.envrc`, and similar) into `/workspace` after seeding, bypassing `.gitignore`. The same flag works on the agent launchers and maps to the config key `box.withEnv` â€” see [configuration](/docs/configuration) for the key and [environment](/docs/environment) for the exact patterns.

```bash
agentbox create --with-env
```

For **arbitrary** host files outside those patterns, use the `carry:` block in `agentbox.yaml` instead â€” see [agentbox.yaml](/docs/agentbox-yaml).

<Callout type="warn" title="HEADS UP">`--with-env` copies secrets into the box, bypassing `.gitignore`. Only use it for boxes you trust with those values. For per-file hostâ†’box copies with a one-time approval prompt, use the `carry:` block â€” see [agentbox.yaml](/docs/agentbox-yaml).</Callout>

## Choosing a branch

By default each box gets a **fresh** per-box branch `agentbox/<box-name>`, forked from the host `HEAD` (bumping to `-2`, `-3`, â€¦ if the name exists, so boxes never collide). Pass `--from-branch <ref>` to fork off a different branch, tag, or SHA, or `-b, --use-branch <name>` to reuse an existing branch directly. See [CLI commands](/docs/cli) for all flags.

```bash
agentbox create --from-branch main
```

Whichever you pick, `git push` from inside the box goes through the host relay with your credentials â€” see [sync and git](/docs/sync-and-git).

<Callout type="warn" title="HEADS UP">`-b/--use-branch` and `--from-branch` are mutually exclusive, and on local Docker `--use-branch` fails if that branch is already checked out on the host.</Callout>
