## Context

A user hit a bug running a big box (`--size ccx23`) on Hetzner. A sizing audit found: Hetzner passes `server_type` through unvalidated and surfaces raw
late API errors (`resource_limit_exceeded`, out-of-stock) after firewall/SSH setup;
location is hardcoded `nbg1` (docs falsely say configurable); every Hetzner box reports
static 2/4/40 resources; Daytona's `--size` is parsed then stripped on the snapshot path
(the SDK only allows resources at snapshot-create time); E2B has no sizing knob at all
(hardcoded 2 vCPU/4096 MB at `Template.build`, though the SDK accepts `cpuCount`/`memoryMB`);
Vercel sizes via a separate `box.vercelVcpus` key while `box.sizeVercel`/`box.sizeE2b` are
dead. Decisions made with the user:

- **Unify everything under `--size` / `box.size<P>` / `box.size`**: hetzner = server type,
  daytona = `cpu-mem-disk` GB (bake-time), e2b = `cpu-mem` GB (bake-time), vercel = vCPU
  int (1/2/4/8). Delete `box.vercelVcpus` (unreleased — clean deletes, no aliases).
- **`agentbox prepare --size`** for the bake-time providers (daytona, e2b); size becomes
  part of the skip-fast check so a changed size rebuilds.
- **No ARM support**: `cax*` types rejected fast with a clear message (base snapshot is x86).
- **Hetzner location**: new `box.hetznerLocation` key (default `nbg1`) + `--location` flag
  on create and prepare.
- Hetzner gets a **preflight validator** over `GET /server_types` + friendly error mapping,
  and reports **real resources** from the create response.

## Phase 2 — Hetzner preflight, error mapping, location, real resources

1. **client.ts**: add `listServerTypes()` (`GET /server_types`, paginated like
   `listServers`) returning `{id, name, cores, memory, disk, architecture: 'x86'|'arm',
   deprecation, prices: [{location}]}` (re-verify exact API field shapes against
   docs.hetzner.cloud during implementation). Add `architecture?` to `HetznerImage`.
2. **New packages/sandbox-hetzner/src/preflight.ts** (pure, no network):
   - `validateServerChoice({serverType, location}, catalog, image)` → throws
     `UserFacingError`, checks in order: type exists (else suggest ~6 valid non-deprecated
     x86 names from the catalog); architecture is x86 (`cax*` → "base snapshots are
     x86-only; ARM types not supported"); not deprecated; `type.disk >= image.disk_size`
     (skip when image null, i.e. stock string refs); location in `type.prices[].location`
     (else list where it IS offered + mention `--location`/`box.hetznerLocation`).
   - `mapHetznerProvisionError(err, choice)`: `resource_limit_exceeded` → account-limits
     explanation + Hetzner Console limits page; `resource_unavailable`/placement → "no
     capacity for <type> in <loc>, try --location fsn1/hel1/ash"; others pass through.
     Always preserve the original message.
3. **backend.ts provision**: after `resolveImageId` (~:392) resolve
   `location = req.location || 'nbg1'`, fetch catalog + `getImage(imageRef)` (numeric refs
   only), run `validateServerChoice` BEFORE firewall/key creation; use `location` in
   `createServer` (:403) and progress line; wrap the createServer call with
   `mapHetznerProvisionError` (outer catch :452 keeps cleanup + rethrows). Return real
   resources from the create response: `{cpu: server.server_type.cores, memory: ...memory,
   disk: ...disk}`.
4. **Real-resources plumbing**: `CloudHandle` (packages/core/src/cloud-backend.ts) and
   `BoxRecord['cloud']` gain optional `resources {cpu, memory, disk}`;
   cloud-provider.ts stores it on the record and logs
   `provisioned <c> vCPU / <m> GB RAM / <d> GB disk` when present. `defaultResources`
   stays as fallback for old records. (Vercel best-effort: `{cpu: vcpus, memory: vcpus*2}`.)
5. **Prepare location**: add `location?` to `PrepareOptions`
   (packages/core/src/provider.ts:285); `prepare.ts` command gains `--location`, resolves
   `opts.location ?? cfg.effective.box.hetznerLocation`, forwards via `runPrepare` (:347);
   `prepareHetznerProvider` (:427) forwards into the existing `location` option.

## Phase 3 — Prepare-time sizing (daytona, e2b)

1. **Plumbing**: add `size?: string` to `PrepareOptions`; `prepare` command gains
   `--size <spec>`; `runPrepare` resolves `opts.size ?? resolveBoxSize(cfg.effective,
   providerName) || undefined` and forwards. Docker/hetzner/vercel ignore it.
2. **Daytona** (packages/sandbox-daytona):
   - prepared-state: `TExtra = {size?: string}`; `preparedMatches(state, sha, size?)`
     requires sha AND size equality (absent size == default bake; NOT folded into

     `cpu-mem-disk` example); snapshot name gains a size suffix
     (`agentbox-base-<12sha>-4-8-20`) so re-sized bakes don't collide; pass
     `resources` into `client.snapshot.create` (:232); persist size in extras.
   - backend.ts snapshot path (:239-245): when `req.size` ≠ baked size, warn loudly via
     `onLog` with the exact `agentbox prepare --provider daytona --size <x> --force` hint.
3. **E2B** (packages/sandbox-e2b):
   - `parseE2bSize('4-8')` → `{cpuCount: 4, memoryMB: 8192}`; a third `-disk` slot is
     accepted-with-warning (E2B SDK has no disk knob); exported for tests.
   - `PrepareE2bOptions.size` wins over the existing `cpuCount`/`memoryMB` defaults
     (:83-84) into `Template.build` (:196); store `size` in prepared state; skip-fast
     (:112-127) requires sha AND size match; `prepareE2bProvider` (:302) forwards.

     resources; E2B rejects them).

## Verification

- `pnpm -w build && pnpm -w test`; `pnpm lint`.
- Config surface: `agentbox config list` shows `box.hetznerLocation`, no `box.vercelVcpus`;
  `config set box.vercelVcpus 4` fails as unknown key.
- Hetzner fast-fail (no server billed): `node apps/cli/dist/index.js create --provider
  hetzner --size cax31 -y -n t1` → ARM message; `--size cx99` → catalog suggestions;
  `--size cx23 --location atlantis` → offered-locations error. All should fail BEFORE
  firewall/SSH-key creation (check ~/.agentbox/logs/create.log ordering).
- Hetzner happy path (background + log-tail, then destroy):
  `node apps/cli/dist/index.js create --provider hetzner --size ccx23 --location fsn1 -y
  -n big-smoke &` then `tail -f ~/.agentbox/logs/create.log`; expect
  `provisioned 4 vCPU / 16 GB RAM / 160 GB disk`; in-box `df -h /` shows ~160 GB
  (settles the growpart question); `agentbox destroy big-smoke -y`.
- Vercel: `create --provider vercel --size 4 -y -n v-smoke`; `--size 3` fails pre-API.
- Daytona: `prepare --provider daytona --size 4-8-20 --force` (snapshot name suffixed);
  `create -y`; `create --size 2-4-8` → bake-time warning; repeat prepare → skip-fast.
- E2B: `prepare --provider e2b --size 4-8 --force`; repeat → skip-fast; `create --size
  2-4` → warning.

## Edge cases / risks

- Existing prepared states have no baked size → treated as default bake; size-less
  prepares still skip-fast; sized prepare rebuilds. No migration (unreleased).
- Old box records lack `cloud.resources` → readers treat as optional.
- Generic `box.size` shared across providers parses differently per provider (hetzner
  preflight/vercel parser reject foreign formats with clear errors); per-provider keys
  are the documented answer for multi-provider projects.
- Behavior change worth a changelog line: claude/codex/opencode, queued jobs, and
  dashboard creates now honor `box.size`/`box.hetznerLocation`.
- Re-verify Hetzner `/server_types` response field shapes (prices[].location name,
  deprecation object) against the live API docs during implementation.
- Confirm Daytona accepts the `-4-8-20` snapshot-name suffix (digits/dashes; current
  names are `agentbox-base-<hex>` so it should).
